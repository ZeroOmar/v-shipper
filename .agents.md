# v-shipper: Docker Volume Migration Application

## Project Overview
A stateless, containerized Python + FastAPI application for managing Docker volumes across multiple pools (local bind mounts and mounted remote storage). Features web UI, sequential migration/backup operations with exclusive locks, and real-time progress tracking.

## Tech Stack
- **Language**: Python 3.11+
- **Framework**: FastAPI (monolithic, REST API + integrated frontend)
- **Container**: Alpine 3.20 Linux (~350-400MB final image)
- **Key Libraries**: 
  - `docker` (Docker SDK for Python)
  - `PyYAML` (config parsing)
  - `uvicorn` (ASGI server)
  - `APScheduler` (optional: cron-based backups)

## Configuration
- **Source**: Multiline YAML in `VOLUME_MANAGER_CONFIG` environment variable
- **Parsing**: Loaded at startup, validated, saved to `/tmp/config.yaml` (for reference)
- **Structure**:
  ```yaml
  docker_hosts:
    - name: host1
      pool: /config/docker_pools/host1
      pool_type: local | remote
  backup_pools:
    - name: backup1
      path: /config/backup_pools/backup1
  web_ui:
    port: 80
    admin_user: admin
    admin_password: <base64>
  ```

## Core Architecture

### Services (`app/services/`)
- **`volume_service.py`**: List/discover/rename/delete volumes, get metadata (size, date, creation), disk stats per pool
- **`migration_service.py`**: Rsync orchestration between pools (preserves permissions/times), lockfile management, post-migration verification
- **`backup_service.py`**: Tar volumes to backup pools, checksum verification
- **`ssh_service.py`**: Deprecated legacy SSH service; not used by current mounted-pool operations
- **`docker_service.py`**: Docker SDK wrapper for volume/container discovery
- **`task_queue.py`**: Sequential in-memory task queue, progress tracking, exclusive locks per volume

### API Routes (`app/api/`)
- `POST /api/login` — Auth with admin credentials
- `GET /api/pools` — List all pools with disk usage stats (total_gb, used_gb, available_gb, usage_percent)
- `GET /api/volumes?pool=<name>` — List volumes in pool
- `GET /api/volume/<pool>/<name>` — Volume metadata (size, date, backups)
- `POST /api/migrate` — Start migration (source pool/volume → dest pool), returns task_id
- `POST /api/backup` — Start backup to backup pool, returns task_id
- `POST /api/rename` — Rename volume
- `POST /api/delete` — Delete volume (warns if no backups)
- `POST /api/pool/create` — Create empty pool (mkdir 777)
- `GET /api/refresh` — Re-read pools from mounted directories
- `GET /api/task/<task_id>/progress` — JSON progress (percentage, current_file, etc.)
- `GET /api/task/<task_id>/logs` — Streaming task logs (stdout only)
- `GET /api/health` — Health check endpoint

### Frontend (`app/templates/` + `app/static/`)
- Simple responsive HTML + vanilla JavaScript + Bootstrap/Tailwind CSS
- Login screen → pool/volume listing with disk usage progress bars
- Migration/backup modals with real-time progress tracking
- No external framework dependency; minimal JS for polling progress

## Key Implementation Details

### Lockfile Strategy
- **Location**: `/tmp/locks/<pool>_<volume_name>.lock`
- **Behavior**: Exclusive locks prevent concurrent operations on same volume
- **Cleanup**: Removed automatically when task completes (success or error)
- **Enforcement**: Task queue checks lock before starting operation

### Rsync for Migration
- **Command**: `rsync -av --perms --preserve-times --group --owner --progress <src> <dst>`
- **Local pools**: Direct rsync between mounted directories
- **Remote pools**: Mounted filesystems treated like local pools for rsync/tar operations
- **Verification**: Optional checksum comparison post-migration
- **Permissions**: `--perms` preserves file permissions/ownership

### Remote Pools
- Remote pools are expected to be mounted on the host running v-shipper
- Operations use standard local filesystem access once the pool is mounted
- No SSH tunneling or remote command execution is performed by the current application

### Sequential Operations
- One migration/backup runs at a time (configured, not scalable)
- Task queue in-memory, processes queue sequentially
- UI polls `/api/task/<task_id>/progress` for real-time updates
- Prevents resource contention, simpler concurrency model

### Disk Space Reporting
- **Local pools**: `os.statvfs(pool_path)` for native FS stats
- **Remote pools**: Mounted filesystems are queried via local filesystem/stat calls
- **Format**: Returns `{total_gb, used_gb, available_gb, usage_percent}`
- **UI**: Progress bars per pool showing usage

### Stateless Design
- No database, no persistent state files on filesystem
- All state in-memory (task queue, progress, locks)
- Container restart = clean slate (no leftover state)
- Config re-read on startup from env variable

## Deployment

### Docker Image
- **Base**: `alpine:3.20`
- **Build**: Multi-stage (build layer + runtime layer)
- **User**: Non-root `appuser` (uid 1000)
- **Exposed**: Port 80 (or configured port)
- **Healthcheck**: `GET /api/health` returns 200
- **Target size**: ~350-400MB

### docker-compose (Local Testing)
- Services: app + mock Docker socket + optional NFS test mount
- Volumes: Local pools, backup pools, config via env
- Network: App accessible on `localhost:8000`

### GitHub Workflow
- **File**: `.github/workflows/docker-publish.yml`
- **Trigger**: Push to main (or tags for releases)
- **Registry**: `ghcr.io/<owner>/v-shipper`
- **Features**: Multi-platform build, image signing via Cosign

## Common Development Tasks

### Adding a New Operation (e.g., clone volume)
1. Add model in `app/models.py` (request/response schemas)
2. Implement logic in `app/services/volume_service.py` or new service
3. Add API endpoint in `app/api/routes.py`
4. Add to task queue in `app/services/task_queue.py` if async
5. Update frontend UI (`app/templates/index.html` + `app/static/main.js`)
6. Test with docker-compose

### Adding Remote Pool Operations
Remote pools should be mounted and accessed as local paths. When adding new remote-pool functionality, treat the mounted pool path like any other local directory and avoid SSH-specific code paths.

### Adding Authentication Method
1. Modify `app/api/routes.py` login endpoint
2. Update config model in `app/config.py` to support new auth scheme
3. Add session middleware if switching from current session model

### Monitoring/Debugging
- All logs printed to stdout (captured by Docker)
- Progress stored in in-memory task dictionary
- Mount path access logged at service level
- Rsync verbosity configurable via `--progress` flag

## Known Constraints
- Sequential operations only (no parallel migrations)
- Configuration secrets should use Docker Secrets or secure environment variables
- No database; state lost on restart (by design)
- Lockfiles are soft locks (cooperative, not enforced at OS level)
- NFS pools must be pre-mounted by Docker (app treats them as local)

## Future Enhancements
- Periodic backup scheduling via APScheduler (cron expressions in config)
- Backup rotation/TTL logic
- Parallel operation mode with resource limits
- WebSocket for real-time progress (vs. polling)
- Metrics export (Prometheus format)
- Multi-user RBAC (vs. single admin account)
